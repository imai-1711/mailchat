MailChat 次世代機能統合に向けた技術レビューと開発ロードマップ



1. 現状のアーキテクチャ俯瞰と技術的負債の再評価



MailChatの「React 18 CDN + Babel Standalone + n8n」というビルド不要の単一HTML構成は、GitHub Pagesへの即時デプロイという俊敏性において極めて強力な武器である。しかし、プロダクトが「AI駆動型メール体験」という次のフェーズへ進むにあたり、現在の1,500行を超えるindex.htmlに蓄積された技術的負債は無視できない段階に達している。



現状の分析



* AttachmentBubbleの限界: 5MB以下のファイルをdataUrlで強引に処理している現状は、モダンなファイルハンドリングとは言い難い。

* 80ms待機ロジックの必然性: モバイル対応等で行われているsetTimeout(80ms)は、ビルド工程を持たない環境下でのReactのステート更新とDOM反映のラグを埋めるための泥臭いが必要不可欠な設計判断である。

* n8n依存の取得フロー: sRUVnMcEkIYzmbCJワークフローによるIMAPポーリングとキャッシュ（StaticData）への依存。



技術的負債の特定と影響



1. P1: 添付キャッシュの不整合 (重要度: 高): StaticDataの古い実態がIMAP同期まで更新されない。これはユーザーが「最新の情報を扱っている」という確信を奪う。

2. P2: 大容量ファイル・CIDリファレンス未対応 (重要度: 高): 5MB制限およびインライン画像（cid:）の欠落。HTMLメールの表示崩れは、プロフェッショナル向けツールとしての信頼性を致命的に損なう。

3. P3: セキュリティと接続の脆弱性 (重要度: 中): CONFIG.CLAUDE_API_KEYの露出リスク、およびワークフロー内のローカルIP（192.168.1.106）直書き。これらは商用レベルの運用において即時修正が必須である。



現状の課題は、単なるバグではなく「AIパワード・コミュニケーションツール」への進化を阻む構造的な壁である。

2. Spikeをベンチマークとした機能ギャップ分析



競合のSpikeが提供する「Conversational UI」と「AIアシスタント」に対し、我々はn8nというオープンなバックエンドを活かした「AIエージェント統合プラットフォーム」としての優位性を確立しなければならない。



機能ギャップと戦略的分析



機能カテゴリ / 現状のギャップ	戦略的評価（So What? / ユーザーへの影響）

Conversational UI / 現状は古典的なスレッド形式	メールの「重さ」を排除し、チャット感覚の即時性を提供。非専門家が「秘書とやり取りする」感覚を得るための必須要件。

AI Summaries / 要約機能未実装	大量メールの海に溺れるユーザーを救う「要約スニペット」こそが、業務効率を劇的に向上させる価値提案の核。

Priority Inbox / 自動仕分け機能なし	重要な連絡を見逃すリスクを低減。n8n側でLLMを用いた高度な優先度判定を実装可能であり、Spike超えの精度を目指すべき。

AI Agent統合 / 基盤はあるが機能未統合	Spikeの閉じたAIに対し、MailChatは個別の「文体（Style-Profile）」を学習した独自のAIエージェントを構築可能。これが最大の破壊的価値となる。

3. P1〜P3既知問題の抜本的解消工程



基盤の脆弱性を解消するための「根本治療」を以下の手順で断行する。



添付ファイル処理とキャッシュの最適化 (P1, P2)



* n8nワークフロー sRUVnMcEkIYzmbCJ の改修: getAttachmentsノードにおいて、バイナリデータをdataUrlに変換する処理を廃止。代わりにバイナリをR2/S3へ即時転送し、フロントエンドには「署名付きURL」を返却するよう構成変更すること。

* キャッシュ整合性: フロントエンドのfetch-emailsリクエスト時に、StaticDataの最終更新日時を検証。不整合検知時はn8n側でIMAP手動フェッチを強制トリガーするロジックを組み込む。



セキュリティと信頼性の強化 (P3)



* Claude APIの隠蔽: フロントエンドからAPIキーを完全排除。n8nに/webhook/claude-proxy（ワークフローID: YiVDaNWrzE8ra73t の拡張または新規作成）を構築し、認証ヘッダーをサーバーサイドで付与。

* インフラの抽象化: キャッシュ保存ノード等の接続先IPを192.168.1.106からhost.docker.internalへ変更、または環境変数化を徹底し、DHCPによる接続断絶を根絶する。

4. 「会話型UI（Conversational Email）」の実装に向けた修正要件



index.htmlを「人中心」のバブル形式UIへ進化させるため、以下のアルゴリズムと処理を実装せよ。



データ構造の変換アルゴリズム



IMAPのヘッダー情報を解析し、メッセージを「会話」へと再定義する。



1. ツリー構築: In-Reply-ToおよびReferencesヘッダーを使用し、メッセージ間の親子関係をマップ化する。

2. フラット化とグルーピング: 構築したツリーを時系列でフラット化。同一送信者（From）が連続する場合、UIバブルを統合。

3. 表示ロジック: 送信者を「人」として扱い、スレッド形式ではなくメッセージアプリのような左右のバブル配置を採用。



CIDレンダリングの技術的解決



ビルドレス環境（NPM利用不可）という制約下でCIDを実現するため、以下の手法を採用する。



* CDNベースのサニタイズ: DOMPurifyをCDNからロードし、dangerouslySetInnerHTMLの安全性を確保。

* 動的リソース置換: DOMParser（ブラウザ標準）を用いてHTML本文をパース。img[src^="cid:"]を抽出し、対応する添付ファイルのdataUrl（またはS3 URL）へ置換した後、DOMを再構成する。

5. AI機能（要約・文体学習）の統合とn8nワークフローの更新



「style-profile」ワークフローをMailChatの心臓部として統合する。



AIインテリジェンスの実装



* 自動要約エンジン: fetch-emails完了後、Claude APIノードにて全件をスキャンし、3行以内の要約をStaticDataにキャッシュ。一覧画面のsnippetとして表示。

* 文体プロファイリング: 送信済みメッセージからユーザー固有の語尾やトーンを抽出し、返信生成プロンプトに動的に注入。



ワークフロー更新の厳格な運用



開発フローにおいて、以下の制約を遵守すること。



* String.raw 禁止: SDK生成時（gen_sdk.py）にエラーを招くため、テンプレートリテラルでのString.raw使用は一切認めない。

* 二重エスケープの徹底: バックスラッシュ（\\xC0等）は、テンプレートリテラル内で正しく評価されるようエスケープ処理を自動化する。

* 更新フローの遵守: _inbox_code.js編集 → fix_workflow.py実行 → SDK生成 → n8n MCPでのパブリッシュ、という一連のパイプラインを死守せよ。

本ロードマップを遂行し、MailChatを「非専門家のための最高峰AIメールツール」へと昇華させる。



優先順位付き実行リスト

  * Claude APIのプロキシ化。

  * 192.168.1.106の参照排除と、添付ファイルキャッシュ更新の自動化。

* 中期（1ヶ月）: UX刷新とAI統合

  * 会話型UI（バブル形式）の完全実装。

  * CIDリファレンス対応によるHTMLメール表示の正常化。

  * AI自動要約および「style-profile」に基づく返信アシストの統合。



本ロードマップの完遂後、MailChatは単なるメールクライアントではなく、ユーザーの意図を汲み取り、文体を模倣し、情報を整理する「パーソナルAIエージェント」へと変貌する。これは既存のクローズドなツールには真似できない、n8n + React 構成だからこそ到達可能な領域である。
